<context>

**Phase goal:** A non-tech server owner can change `max-players`, `gamemode`, `pvp`,
`difficulty`, `motd`, `white-list`, and a curated set of other `server.properties`
options through labeled, typed inputs with plain-English descriptions, see exactly
what will change before saving, and roll back to any previous version if they regret
it - without ever touching raw text or risking a typo that breaks the server.

**Origin note:** `.planning/notes/friendlier-configs-tab.md`

**Hard rules from `CLAUDE.md`:**
- English only. New UI strings go into `i18n.json` under `en.dictionaries.en.configs.*`.
  Do **not** add Spanish strings (the existing `es` block is pre-existing legacy
  content; do not expand it).
- No new npm dependencies. Stay within React 18 + Tailwind + shadcn/ui (Radix) +
  Sonner + the existing dep set listed in `package.json`.
- Path-traversal guards in `safeResolve` / `resolveEditable` must be reused; no new
  filesystem route should bypass them.
- Do **not** commit or push unless the user explicitly asks.
- The `CLAUDE.md` file is out of date about the frontend (still says "plain
  HTML/CSS/JS"); the real frontend is React + Vite + Tailwind. Trust the code in
  `src/`, not the description in `CLAUDE.md`.

**Verification approach:** the repo has no test framework. "Verify" means starting
the panel with `npm start`, opening `http://localhost:2121`, and exercising the
feature. Document manual verification steps in the SUMMARY so the user can repeat.
</context>

<objective>

Rewrite the Configs tab so a non-tech user can change `server.properties` settings
through labeled, typed controls with plain-English descriptions, preview a diff
before saving, see a "restart required" banner with a one-click restart, browse
files via a grouped left rail, and roll back to any previous `.bak` snapshot.

Scope (from `.planning/notes/friendlier-configs-tab.md`, items 1, 3, 4, 5, 6, 8):

- **Item 1** Friendly forms for `server.properties` (typed inputs, labels,
  descriptions, "Advanced keys" disclosure for unmodeled keys).
- **Item 3** Live validation (range/enum/parse) with a yellow "Issues" panel.
- **Item 4** Diff preview before save (green/red, no new deps).
- **Item 5** "Requires restart" banner after save with one-click Restart.
- **Item 6** Grouped file nav (left rail: Gameplay / Performance / World / Other).
- **Item 8** `.bak` history dropdown + restore (two new backend endpoints).

**Explicitly out of scope (Phase 2/3):** syntax highlighting in raw view, per-key
search, presets, first-run wizard.

**YAML files** (`spigot.yml`, `bukkit.yml`, `paper-world-defaults.yml`, etc.) keep
the existing raw monospace editor in this phase. The only YAML behavior added is:
the diff preview, validation that the file still parses as YAML before save, and
the restart banner. No friendly forms for YAML in Phase 1.
</objective>

<must_haves>

1. Non-tech user can change `max-players`, `gamemode`, `difficulty`, `pvp`, `motd`,
   `white-list`, `allow-flight`, `hardcore`, `spawn-protection`, `view-distance`,
   `simulation-distance`, `server-port`, `online-mode`, `enable-command-block`,
   `level-name`, `level-seed`, `level-type`, `max-tick-time`, `max-world-size`,
   `generate-structures` through friendly controls with plain-English labels and
   descriptions.
2. Unknown `server.properties` keys collapse into an "Advanced keys" disclosure
   that still edits the raw value.
3. Validation runs on every change: numeric ranges, enum membership, integer
   parsing, YAML parseability. A yellow "Issues" panel lists problems; save is
   disabled while any blocking issue is present.
4. A diff preview shows green/red line changes before save. Saving only happens
   after the user confirms.
5. After a successful save, a persistent banner across the top of the card says
   "You saved `{file}`. Restart the active server to apply." with a Restart button
   that calls the existing `restart` action and a Dismiss button.
6. The file picker becomes a left rail with groups: Gameplay, Performance, World,
   Other. Each group shows the files in that group; the active file is highlighted.
7. A "History" dropdown in the card header lists recent `.bak` files (newest
   first) with relative timestamps and size. Clicking one opens a confirm dialog
   explaining the restore will also create a fresh `.bak` of the current state.
8. Restoring a `.bak` puts the chosen backup's content back as the current file
   *and* creates a fresh `.bak` of the state that was just overwritten (so users
   can undo the restore).
9. New backend endpoints `GET /api/configs/:name/backups` and
   `POST /api/configs/:name/restore` are JWT-protected, only operate on
   allowlisted files, and refuse any name that escapes the server folder.
10. All new UI strings are added under `en.dictionaries.en.configs.*` in
    `i18n.json`. No Spanish strings are added.

</must_haves>

<task type="auto" name="1.2 Add .bak restore endpoint">

In `server.js`, add a JWT-protected route:

```js
app.post('/api/configs/:name/restore', auth, (req, res) => { ... });
```

- Body: `{ backup: '<name>.bak' }`.
- Resolve the server folder the same way the existing `GET /api/configs/:name`
  PUT route does (read it first; match its allowlist + path-safety pattern).
- Verify the backup file exists inside the server folder and matches the
  pattern `^<basename>\.\d{8}-\d{6}\.bak$`. Reject anything else with
  `{ error: 'invalidBackup' }`.
- Read the current file content. Write a fresh timestamped `.bak` of the current
  state (mirror the existing PUT save logic so the on-disk `.bak` shape matches).
- Copy the backup file's content over the current file. Return
  `{ ok: true, content }` so the frontend can refresh.
- Use the same auth + error shape as the other `/api/configs/*` routes.

Commit atomically: `feat(configs): restore config from .bak snapshot`.

</task>

<task type="auto" name="1.4 Config file parsing helpers">

Create `src/lib/configFile.js`:

- `parseProperties(text)` → `{ order: string[], values: Record<string,string>,
  comments: Record<string,string[]> }`. Preserve the order of keys as they
  appear in the file (the friendly form needs this to round-trip without
  reordering). Lines starting with `#` and blank lines are treated as comments
  attached to the following key. Values keep their trailing whitespace and any
  inline comment after `#` is preserved as part of the comment map.

  stable for files the form does not touch.
- `isYamlFilename(name)` → boolean. Returns `true` for `*.yml` and `*.yaml`.
- `isPropertiesFilename(name)` → boolean. Returns `true` for
  `server.properties` (case-insensitive). Extend later if more `.properties`
  files appear.
- `validateValue(schemaKey, value)` → `{ ok: boolean, error?: string }` using
  the schema defined in 1.5.
