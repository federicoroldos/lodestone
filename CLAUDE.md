## What this is

**Lodestone** is a self-hosted web panel for managing Minecraft servers (Spigot / Paper /
Vanilla / Fabric / Forge / NeoForge) on **Windows, Linux, or macOS**. It is a single Node.js
process that serves a REST API + WebSocket and a single-page frontend.

- Backend: **`server.js`** (one file, CommonJS).
- Frontend source: **`src/`** — **React + Vite**, Tailwind CSS, Radix UI primitives
  (`src/App.jsx`, `src/main.jsx`, `src/views/*`, `src/components/*`). i18n strings live in
  **`i18n.json`** (en + es parity required).
- Frontend build output: **`public/`** — `public/index.html` + `public/assets/index-*.{js,css}`
  are **generated by `vite build`**. `server.js` serves `public/` statically. **Do not edit
  `public/assets/*` by hand** (there is no longer a `public/app.js`); edit `src/` and rebuild.
- Settings + state: **`config.json`** (git-ignored), **`metrics.json`** (git-ignored).
- Launcher: **`start-panel.bat`** (installs deps on first run, then `node server.js`).

Default URL: `http://localhost:2121`. Node **18+** required (uses the global `fetch`).

## Run / develop

```bash
npm install      # first time only
npm start        # node server.js — serves the API + the built public/ (no watch)
npm run dev      # vite dev server with HMR for editing src/ against a running backend
npm run build    # vite build — regenerates public/ from src/ (commit the output)
```

There are **no tests and no linter**. "Verify" means: start the panel, open
`http://localhost:2121`, log in, and exercise the affected tab.

- **Frontend changes (`src/`) require a `npm run build`** before `npm start` reflects them —
  the served `public/` is build output, not source. (Use `npm run dev` for live editing.)
  When you change `src/`, rebuild so `public/assets/*` and `public/index.html` stay in sync,
  and commit the regenerated assets alongside the source.
- **Backend changes (`server.js`) need a panel restart** (no watch).

## Architecture

### Multi-server model
- `config.servers[]` holds registered servers: `{ id, name, dir, jar, javaArgs, mcVersion,
  stopTimeoutSeconds, worlds, watchdog }`.
- One **`ServerManager`** instance per server, kept in the `managers` Map. Each wraps a
  `child_process.spawn('java', ...)` (no shell, so spaces and non-ASCII paths work) and tracks
  status, console history, players, TPS, and the watchdog.
- `config.activeServerId` is the default target for most views. Endpoints also accept
  `?serverId=` (query) or `serverId` (body) to override — see `targetManager(req)`.
- Status flows: `offline → starting → online → stopping → offline`. "online" is detected by
  the `Done (Xs)!` line; players/TPS are parsed out of console lines and refreshed by polling
  `list` / `tps` on an interval.

### Auth
- Users live in `config.users[]` as `{ id, username, name, passwordHash }` (an optional
  `email` field is reserved for a future opt-in feature and is not used for login). Passwords
  are hashed with **scrypt** (`salt:hash`), compared with `timingSafeEqual`.
- Login (`POST /api/login`) returns a **JWT** signed with `config.jwtSecret`. All `/api/*`
  routes except `/api/login` require `Authorization: Bearer <token>` (or `?token=` for
  downloads / the WebSocket). `userFromToken` re-resolves the live user, so deleting a user
  immediately invalidates their sessions.
- Any logged-in user can manage users and servers (no role system).

### WebSocket (`/ws`)
- Token-checked on upgrade. On connect the client gets a `meta` frame, a `status` frame per
  server, and the active server's console `history`. Live frames: `line`, `status`, `stats`.
- `globalBroadcast` fans out to all clients. Resource `stats` for the active server are pushed
  every 2 s; per-minute `metrics` samples are persisted to `metrics.json` (7-day retention).

### Config persistence
- `loadConfig()` reads `config.json`; `saveConfig(next)` writes the whole object back
  (pretty-printed). **Any mutation of `config` must be followed by `saveConfig(config)`** to
  persist. `migrateConfig()` upgrades legacy single-server / single-password configs on boot.

## Frontend (`src/` — React + Vite)

Edit the React source under `src/`, never the built `public/assets/*` (see "Run / develop").

- **`src/main.jsx` / `src/App.jsx`** bootstrap the SPA and route between views via the sidebar.
  One file per view in **`src/views/*`** (`DashboardView, ServersView, MetricsView, ConsoleView,
  PlayersView, MapView, PluginsView, ModrinthView, FileManagerView, ConfigsView, BackupsView,
  TasksView (Schedules), UsersView, LoginView`).
- **State/context** lives in `src/context/*` (`AuthContext`, `ServerContext`, `I18nContext`)
  and **`src/hooks/*`** (`useApi`, `useApiStream`, `useWebSocket`): the token sits in
  `localStorage`, the API hooks attach the bearer token, `useWebSocket` is the live client.
- **Components:** reusable UI in `src/components/ui/*` (Radix + Tailwind primitives — `button`,
  `dialog`, `select`, …), shared widgets in `src/components/shared/*` (`ConfirmDialog`,
  `PromptDialog`, `StatusPill`, `BrandMark`, …), layout in `src/components/layout/*`. **Reuse
  these primitives** rather than hand-rolling new markup/styling.
- **Styling** is Tailwind (`src/index.css`) + `tailwind-merge`/`cva`; toasts via `sonner`.
- **i18n:** every user-facing string goes through the i18n layer (`src/i18n`, `src/context/
  I18nContext`) and must be added to **`i18n.json` in both `en` and `es`** (parity is enforced
  by convention — the English-only rule governs code/comments/logs, not the user-facing `es`
  translations).

### Server creator — version sources (no hardcoded version tables)

Each server type in the creator lists its installable Minecraft versions **dynamically from
the upstream source**, so the list stays correct as new releases ship with **zero manual
maintenance**. Never hardcode a table of versions (or a MC→loader-version map) that a human
has to edit when a new release lands — derive it from the source instead. The pattern lives in
`server.js` (`listServerVersions(type)` → `resolveServerJar(type, mc)`):

| Type | Source | How versions are derived |
| --- | --- | --- |
| vanilla | community gist | parse the MD table, full releases only |
| paper | Fill v3 API | flatten `versions` families, full releases only |
| spigot | getbukkit.org | scrape the download page |
| fabric | Fabric meta API | `versions/game` filtered to `stable` |
| forge | Forge Maven XML | derive the MC version from each coordinate's prefix |
| neoforge | NeoForge releases API (`/api/maven/versions/releases/...`) | derive the MC version from each build via `mcVersionFromNeoForge`, **stable releases only** (`isStableNeoForge`) |

When adding a new type or source: filter to **stable, non-snapshot** releases (no
rc/pre/beta/alpha), sort **newest-first**, and reuse the shared installer plumbing
(`runForgeInstaller` / `findForgeServerJar`) where it applies. If upstream uses a non-obvious
version scheme, mirror the algorithm the project's own site/API uses rather than inventing one
(NeoForge's MC mapping was copied from neoforged.net's download page).

## Important guardrails (don't loosen without understanding them)

- **Path traversal:** the file manager and configs editor resolve user paths through
  `safeResolve(root, rel)` / `resolveEditable(...)`, which refuse anything escaping the server
  folder. Keep every new filesystem route behind one of these.
- **Upload filters:** plugin upload only accepts `.jar`; the config editor only allows
  `server.properties` and `.yml`/`.yaml`. The general file manager allows text files up to
  2 MB for editing (see `TEXT_EXTS`, `MAX_EDIT_BYTES`).
- **Modrinth install** verifies the downloaded version's loader and MC version against the
  server's detected compat (`detectCompat`) before writing it — don't bypass that check.
- **Stopping the panel never stops the Minecraft servers** (intentional, so a panel restart
  doesn't drop players). `shutdown()` leaves running children alive.
- **Backups** run `save-off` + `save-all flush` before zipping an online server and `save-on`
  after; `backupInProgress` guards against overlap.

## Conventions

- Two-space indent, single quotes, semicolons, `'use strict'`, CommonJS `require`.
- Endpoints return `{ ok: true, ... }` on success or `{ error: 'message' }` with a 4xx/5xx
  status. User-facing error strings are short and plain-English.
- Use `log(...)` (timestamped) for panel logs; per-server console lines go through
  `m.pushLine(text, level)` where level ∈ `info | warn | error | cmd`.
- IDs are `crypto.randomUUID()`; user-facing folder names go through `slugify()`.
- **No hardcoded data that the world keeps changing.** Version lists, download URLs derived
  from versions, loader mappings, etc. must be fetched/derived from upstream at runtime — never
  baked into a constant a human has to update (see "Server creator — version sources"). A small
  static constant for genuinely fixed facts is fine; a list that grows with every upstream
  release is not.
